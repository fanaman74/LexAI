# LexAIv2 v1.1 — Semantic Search, Chat-with-File, UI Refresh — Design

**Date:** 2026-06-10
**Status:** User delegated decisions ("proceed without my input"); decisions below are final.

## Goals

1. Semantic search over converted Markdown using chunking + vector embeddings
   stored in SQLite (vector-DB capability via sqlite-vec).
2. Chat with one or more files (multi-turn), launchable from search results
   and the document view.
3. UI refresh: bubble (pill) filters, open-in-Finder button, per-file
   MD/Original view buttons, general visual polish.

## Decisions

- **Embeddings provider:** OpenRouter `POST /api/v1/embeddings`
  (OpenAI-compatible), reusing `OPENROUTER_API_KEY`. Default model
  `openai/text-embedding-3-small` (1536 dims), overridable via
  `EMBEDDING_MODEL` env var. Chosen by user over a local model.
- **Vector storage:** `sqlite-vec` Python package; `vec0` virtual table
  `chunk_vectors(embedding float[N])` keyed by chunk id. If the sqlite3
  build refuses loadable extensions, fall back transparently to numpy
  brute-force cosine over `chunks.embedding` BLOBs. Both paths behind one
  `vectors.py` interface: `upsert(chunk_id, vec)`, `search(vec, k)`,
  `delete_for_file(file_id)`.
- **Chunking:** split Markdown at paragraph boundaries into ~1500-char
  chunks with 200-char overlap; stored in new `chunks` table at conversion
  time (and re-chunked on re-conversion). Embedding happens in a separate
  indexing pass (needs network): after each scan finishes, and via
  `POST /api/index` (re)index endpoint with progress; files whose chunks
  lack embeddings are picked up automatically.

## Schema additions

- `chunks` — `id`, `file_id` (FK, CASCADE), `chunk_index`, `text`,
  `embedding` BLOB NULL (float32 array bytes; NULL = not yet indexed),
  UNIQUE(file_id, chunk_index).
- `chats` — `id`, `file_ids` (JSON), `title`, `created_at`.
- `chat_messages` — `id`, `chat_id` (FK, CASCADE), `role`
  (`user|assistant`), `content`, `created_at`.
- vec0 virtual table `chunk_vectors` (when sqlite-vec loads).

## Backend

- `chunking.py` — `chunk_markdown(md) -> list[str]` (pure, tested).
- `embeddings.py` — `embed_texts(list[str]) -> list[list[float]]` via
  OpenRouter, batched (max 96 per call); raises `EmbeddingError` with a
  clear message when key missing/HTTP error.
- `vectors.py` — storage/search abstraction described above.
- `indexer.py` — background index job (thread, JOBS-style progress like
  ingest): finds chunks with NULL embedding, embeds, stores. Triggered by
  scan completion and `POST /api/index`.
- Routes:
  - `POST /api/index`, `GET /api/index/status` — run/poll indexing.
  - `GET /api/semantic-search?q=` — embed query, top-12 chunks, grouped by
    file: `{file_id, original_name, file_type, score, snippet}` (best
    chunk text trimmed ~300 chars). 400 if not indexed/key missing.
  - `POST /api/chats {file_ids, message}` — creates chat (title = first
    40 chars of message), calls OpenRouter chat with system prompt + full
    MD of files (40k chars/file cap) + message; stores both messages.
  - `POST /api/chats/{id}/messages {message}` — continues chat with full
    history.
  - `GET /api/chats`, `GET /api/chats/{id}` — list/detail.
  - `POST /api/files/{id}/reveal {location_index}` — `open -R` the file's
    path in Finder (404 unknown file, 400 bad index; no-op result if path
    no longer exists, reported as `{"ok": false, "error": ...}`).
  - `GET /api/files/{id}/original?inline=1` — serve with real MIME type
    (pdf → application/pdf, txt/csv → text/plain, etc.) and inline
    disposition so the browser can preview.
- Ingest change: `save_markdown` path also rewrites `chunks` rows
  (embedding NULL); scan completion kicks the indexer if the API key is
  set (silently skipped otherwise).

## Frontend

- **Library:** sidebar replaced by horizontal bubble-filter bar — chip
  groups for Folders, Type, Status, Tags; multi-select; active chips
  highlighted (indigo); counts in chips. Search box gains a
  Keyword ⇄ Semantic toggle (pill). Semantic results render as cards
  (name, score %, snippet, buttons: Open file → document view,
  Chat → /chat?ids=). Each table row gains action buttons:
  **MD** (document view, markdown tab), **Original** (opens
  `?inline=1` in new tab), **📂** (reveal in Finder), **Chat**.
- **Document view:** tab bar Markdown | Original (PDF/text in iframe via
  inline endpoint; other types: download card). Reveal-in-Finder button
  per location. Chat button in header.
- **Chat page (`/chat`):** file chips for context, message thread
  (user right / assistant left, Markdown-rendered), input box, Enter to
  send, chat history sidebar (recent chats, click to resume).
- **Polish:** indigo accent palette, status pills (green/red/amber),
  consistent cards/shadows/rounded corners, loading and empty states,
  sticky top nav.

## Error handling

- Semantic search without key or before indexing → friendly inline
  message with an "Index now" button (and key instructions).
- Embedding API failures mark the index job failed with the HTTP error;
  chunks stay NULL and are retried on next index run.
- Chat errors keep the typed message in the box.

## Testing

pytest TDD: chunking (sizes, overlap, paragraph integrity), vectors
(store/search roundtrip + fallback path), indexer (mocked embeddings),
semantic-search endpoint (mocked), chats endpoints (mocked OpenRouter),
reveal endpoint (mocked subprocess), inline MIME. Frontend: build passes;
manual smoke.

## Out of scope

Streaming chat responses, citations with chunk-level deep links,
re-ranking, multi-user.
